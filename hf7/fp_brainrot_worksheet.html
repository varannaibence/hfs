λ
Lambda Lali
LAMBDA / NÉVTELEN FÜGGVÉNY  ·  \x -> ...

Lali névtelen. Senkinek nem mutatkozik be — csak csinálja a dolgát. Egy rövid, egyszer használatos függvény, amit nem érdemes külön megnevezni. A funkcionális programozás lelke.

"Nincs nevem. Van feladatom."
-- névtelen függvény Haskell-ben: \x -> x * 2 -- alkalmazva: map (\x -> x + 1) [1,2,3] -- → [2,3,4] -- lambda változóhoz kötve: let ketszerez = \x -> x * 2
💡 LAMBDA KALKULUS
Az egész funkcionális paradigma a lambda kalkuluson alapul (Church, 1930-as évek).
🔁
Redukciós Réka
REDUKCIÓS RENDSZER · REDUKÁLÁS  ·  →β

Réka egyszerűsít. Egy kifejezést addig cserél egyenértékűbb, de egyszerűbb alakokra, amíg már tovább nem lehet. Ez a kiértékelés lényege: lépésről lépésre redukálni, amíg el nem érjük a normál formát.

"Minden lépéssel közelebb a végső igazsághoz."
-- redukciós lépések (β-redukció): (\x -> x + 1) 5 -- ↓ x helyére 5 kerül 5 + 1 -- ↓ kiértékelés 6 -- normál forma! -- α-redukció: változók átnevezése -- η-redukció: \x -> f x ≡ f
💡 REDUKCIÓS SZABÁLYOK
β: argumentum behelyettesítés  |  α: átnevezés  |  η: egyszerűsítés
✅
Normál Forma Norbert
NORMAL FORM  ·  nem redukálható tovább

Norbert a végállomás. Egy kifejezés normál formában van, ha már egyetlen redukciós lépéssel sem egyszerűsíthető tovább. Nem minden kifejezésnek van normál formája — van, ami örökké redukálódik.

"Tovább már nem megyek. Ez az igazság."
-- Normál formák: 42 -- ✓ szám, tovább nem megy "hello" -- ✓ string literál True -- ✓ Bool konstruktor -- NEM normál forma (redukálható): 2 + 3 -- → 5 length [1,2] -- → 2 -- Végtelen (nincs normál formája!): let x = x in x -- ∞ loop
⚠ CHURCH-ROSSER TÉTEL
Ha egy kifejezésnek van normál formája, az egyedi — a sorrend nem számít.
🔗
Függvényösszetétel Fanni
FUNCTION COMPOSITION  ·  (f ∘ g)(x) = f(g(x))

Fanni összekapcsol függvényeket. Az egyik kimenetét a másik bemenetére köti — mint csöveket egymáshoz. Az eredmény egy új függvény, ami mindkét lépést egyszerre csinálja.

"Én csak közvetítek g és f között. Az eredmény az övék."
-- Haskell-ben a (.) operátor: dupla = (*2) negyel = (+4) dupla_majd_negyel = negyel . dupla -- (negyel . dupla)(3) -- = negyel(dupla(3)) -- = negyel(6) = 10 dupla_majd_negyel 3 -- → 10
💡 PIPELINE
A |> (pipe) operátor soknyelvben ugyanez: 3 |> dupla |> negyel
🌀
Rekurzív Rita
REKURZIÓ FP-BEN  ·  nincs loop, csak önhívás

A funkcionális paradigmában nincs for/while ciklus. Rita az egyetlen ismétlési eszköz: a rekurzió. Minden iteratív algoritmus kifejezhető rekurzívan. A base case a kulcs — nélküle Rita örökre szól.

"Ciklus? Soha nem hallottam. Van viszont önmagam."
-- iteratív helyett rekurzív összeg: osszeg [] = 0 -- base case osszeg (x:xs) = x + osszeg xs osszeg [1,2,3] -- = 1 + osszeg [2,3] -- = 1 + 2 + osszeg [3] -- = 1 + 2 + 3 + 0 = 6
💡 MINTAILLESZTÉSSEL EGYÜTT
FP-ben a rekurzió és a mintaillesztés szinte mindig kéz a kézben jár.
🐢
Tail-Rekurzív Tünde
TAIL RECURSION  ·  az utolsó hívás maga a rekurzió

Tünde okosabb Ritánál: a rekurzív hívás az utolsó dolog, amit csinál — semmi nem várja az eredményét visszafelé. Ezért a fordító nem halmoz stack frame-eket — helyette csak frissíti a paramétert. Stack Overflow Stan nem tud belépni.

"Ha én vagyok az utolsó, nincs miért várakozni."
-- NEM tail-recursive (vár visszafelé): osszeg (x:xs) = x + osszeg xs -- ^ ezt még ki kell számolni! -- TAIL recursive (akkumulátor): osszegT [] acc = acc osszegT (x:xs) acc = osszegT xs (acc + x) -- utolsó lépés! osszegT [1,2,3] 0 -- → 6
💡 TCO
Tail Call Optimization: sok fordító automatikusan iterációvá alakítja.
😴
Lusta László
LAZY EVALUATION  ·  call-by-need

László nem számol, amíg nem muszáj. Ha egy kifejezésre nincs szükség, el sem végzi. Ez lehetővé teszi a végtelen adatszerkezeteket is — László csak annyit számol belőlük, amennyit tényleg kérnek.

"Miért számolnék olyat, amit úgyse kérsz?"
-- Haskell alapból lusta! -- Végtelen lista — László nyugodt: let vegtelenSzamok = [1..] -- Csak az első 5-öt kéri → csak azt számolja: take 5 vegtelenSzamok -- → [1,2,3,4,5] (nem számolja a többi millió elemet) -- Fibonacci végtelen sorozat: fibs = 0 : 1 : zipWith (+) fibs (tail fibs)
⚠ VESZÉLY
László néha kiszámíthatatlan memóriát és lassulást okoz (thunk halmozódás).
🦅
Mohó Mónika
EAGER EVALUATION  ·  call-by-value

Mónika azonnal kiszámol mindent, amint látja. Nem vár kérésre — ha van egy kifejezés, máris fut. Kiszámítható és gyors, de végtelen listával lefagy.

"Látom, számolom. Várni? Az nem az én stílusom."
-- Python, JavaScript, ML = mohó -- Haskellben: seq vagy BangPatterns kényszeríti -- Mohó Mónikánál: f(g(3)) -- ELŐSZÖR g(3) fut le -- AZTÁN f(eredmény) -- Végtelen lista? Mónika meghal: -- take 5 [1..] → ∞ loop mohó nyelvben
💡 LAZY vs EAGER
Lusta: call-by-need. Mohó: call-by-value. Strict: call-by-value + kiértékeli hamarabb.
💥
Mellékhatás Máté
SIDE EFFECT  ·  a tiszta FP ellensége

Máté olyat csinál, amit nem mondott. Egy függvény mellékhatásos, ha visszatérési értékén kívül megváltoztat valamit a külvilágban: fájlt ír, képernyőre nyomtat, globális változót módosít, hálózathoz nyúl. A tiszta FP kerüli Mátét.

"Senki nem kért, hogy ezt csináljam. De megtettem."
-- Mellékhatás: konzolra írás putStrLn "Hello" -- IO mellékhatás -- Mellékhatás: fájlba írás, globális változó -- Haskellben Máté csak IO monadban élhet: main :: IO () main = do putStrLn "IO-ban vagyok" -- itt Máté megjelenhet -- Tiszta függvény: NINCS mellékhatás negyel x = x * 4 -- csak visszatér
💡 IO MONAD
Haskellben Máté "be van zárva" az IO monadba — nem szivárog tiszta kódba.
🪞
Hivatkozási Hanna
REFERENTIAL TRANSPARENCY  ·  helyfüggetlenség

Hanna bárhol ugyanaz. Ha egy kifejezés hivatkozásilag átlátható, értéke nem függ attól, a program melyik részén áll. Helyettesíthető az értékével, és a program viselkedése nem változik. Nincs változó — csak konstans és nevesített konstans.

"Akárhol olvasol, ugyanaz vagyok. Nincs kontextus, ami megvált."
-- Hivatkozásilag átlátható: let x = 2 + 3 -- x bárhol 5, mindig 5 -- x + x == 10 ↔ (2+3) + (2+3) == 10 ✓ -- NEM átlátható (pl. C++ globális változó): -- int g = 0; -- int f() { return g++; } -- más értéket ad vissza! -- Haskellben minden függvény átlátható: ketszerez 5 -- mindig 10, bárhol
💡 NINCS VÁLTOZÓ
Tiszta FP nyelvben nincs változó — csak konstansok és nevesített konstansok (let/where).
🏗️
Higher-Order Hugó
HIGHER ORDER FUNCTIONS  ·  map, filter, fold

Hugó függvényeket vesz át és/vagy ad vissza — tehát a függvény nála elsőrendű polgár, simán átadható paraméterként. A map, filter, fold mind Hugó munkái.

"Nekem add a függvényt. Én majd tudok vele mit kezdeni."
-- map: alkalmaz minden elemre map (*2) [1,2,3] -- → [2,4,6] -- filter: szűr feltétel szerint filter even [1..6] -- → [2,4,6] -- függvényt visszaad: szoroz n = (*n) -- → függvényt ad let haromszoroz = szoroz 3 haromszoroz 7 -- → 21
💡 FIRST CLASS CITIZEN
Ha egy nyelvben a függvény "first class citizen", bármit lehet vele, mint adattal.
🍛
Curry Csaba
CURRYING  ·  f(a,b) → f(a)(b)

Csaba nem kér mindent egyszerre. Egy többparaméteres függvényt átalakít egymásba ágyazott, egyparaméteres függvények sorozatává. Ha csak egy paramétert adsz, részlegesen alkalmazott függvényt kapsz vissza.

"Adj egyet. Kapsz egy függvényt. Adj még egyet. Kapsz eredményt."
-- Haskellben MINDEN függvény curryzett: add :: Int -> Int -> Int add x y = x + y -- részleges alkalmazás: let add5 = add 5 -- Int -> Int add5 3 -- → 8 -- add 5 3 == (add 5) 3 == add5 3 == 8 map (add 10) [1,2,3] -- → [11,12,13]
💡 HASKELL CURRY UTÁN
A Haskell programozási nyelvet Haskell Curry matematikusról nevezték el.
🎯
Mintaillesztő Miska
PATTERN MATCHING  ·  strukturális overload

Miska az adatok formájára figyel. Ahelyett hogy if-else láncot írna, a függvény különböző "mintákra" különbözőképp reagál. Ez az FP-s overload: nem a típus, hanem a struktúra dönti el, melyik ág fut le.

"Mutasd az alakodat. Akkor tudom, mit tegyek veled."
leiras :: [Int] -> String leiras [] = "üres lista" leiras [x] = "egy elem" leiras [x, y] = "két elem" leiras (x:_) = "több mint kettő" leiras [] -- → "üres lista" leiras [1,2,3] -- → "több mint kettő"
💡 GUARD-OK
Mintaillesztés kiegészíthető guard-okkal: | x > 0 = "pozitív"
🧊
Immutable Imre
IMMUTABILITY  ·  értékek nem változnak

Imre soha nem változik. Ha egyszer lett egy értéke, az az övé örökre. Nincs Változó Vlad felülírás — ha "új" értéket akarsz, új konstanst hozol létre. Imre teszi lehetővé a hivatkozási átláthatóságot.

"Amit kaptam, az vagyok. Senki nem veheti el."
-- Haskellben MINDEN immutable: let x = 5 -- x = 6 → FORDÍTÁSI HIBA -- "módosítás" = új érték létrehozása: let lista = [1,2,3] let lista2 = 0 : lista -- [0,1,2,3] -- lista változatlan: [1,2,3] -- Record update syntax (új rekord): let k2 = k { kor = 4 } -- k megmarad
💡 THREAD SAFETY
Immutable adatok race condition és deadlock nélkül oszthatók szálak között.
📌
Változó Vazul
VÁLTOZÓK FP-BEN  ·  csak konstansok léteznek

Vazul más, mint Vol. 1 Vlad. FP-ben nincs újraírható változó — csak konstansok (fix értékek) és nevesített konstansok (let/where kötések), amik egyszer kapnak értéket és soha nem változnak.

"Vlad cserélgethet. Én? Én vagyok, aki vagyok."
-- FP-ben ez NEM változó, hanem kötés: let n = 42 -- nevesített konstans -- where-ben is: kerulet r = 2 * pi * r where pi = 3.14159 -- lokális konstans -- Nincs ez: -- n = n + 1 → ÉRTELMETLEN (körkörös def.) -- Állapot? → rekurzív paraméterben tárolódik
💡 MIÉRT JÓ?
Ha nincs változó, nem kell nyomon követni, mit változtatott meg ki. Könnyebb tesztelni.
😇😈
Tiszta Tivadar & Nem Tiszta Nóra
PURE vs IMPURE FUNCTIONS / LANGUAGES

Tivadar (pure): ugyanarra a bemenetre mindig ugyanazt adja vissza, és semmilyen mellékhatása nincs. Nóra (impure): néha máshogy viselkedik, olvas/ír, vagy globális állapotot változtat.

"Tivadar: 'Adj 5-öt, kapsz 25-öt. Mindig.' — Nóra: 'Attól függ.'"
-- PURE (Tivadar): determinisztikus, mellékhatás nélkül negyzetEl x = x * x -- mindig x² -- IMPURE (Nóra): IO, random, globális állapot getLine -- más értéket adhat vissza randomRIO (1,6) -- nem determinisztikus -- Tiszta NYELVEK: Haskell (IO monadba zárja Nórát) -- Nem tiszta: Python, JavaScript, Erlang, Scala
💡 HATÁR
Haskell nem tiltja Nórát — csak elkülöníti. IO típus jelzi: "itt Nóra van."
📜
Term Újraíró Tamás
TERM REWRITING SYSTEMS  ·  l → r szabályok

Tamás szabályokat alkalmaz szimbólikus kifejezésekre. Ha egy kifejezés illeszkedik egy bal oldalra (l), lecseréli a jobb oldalra (r). A redukciós rendszer formalizált változata — az FP kiértékelés elméleti alapja.

"Ha l-t látok, r-t írok. Nincs miért, így van megírva."
-- Term újraírási szabályok (elméleti jelölés): -- ADD(0, x) → x -- ADD(S(x),y) → S(ADD(x, y)) -- Példa lépések: -- ADD(S(0), S(0)) -- → S(ADD(0, S(0))) [2. szabály] -- → S(S(0)) [1. szabály] -- = 2 ✓ -- Haskell mintaillesztés = term újraírás! add 0 y = y add (n+1) y = 1 + add n y
💡 CONFLUENCE
Egy TRS "konfluens", ha a szabályok sorrendjétől független az eredmény.
🕸️
Gráf Újraíró Gábor
GRAPH REWRITING SYSTEMS  ·  megosztott struktúrák

Gábor olyan, mint Tamás, de gráf struktúrán dolgozik — nem fán. Azért, mert a gráf megosztott csomópontokat engedélyez: ha kétszer ugyanazt a kifejezést használod, nem másolod, hanem rámutat mindkettő. Ezzel a lusta kiértékelés hatékonyan megvalósítható.

"Tamás fán dolgozik. Én gráfon. Én nem másolom a csomópontokat."
-- Fa (term): x+x kétszer tárolja x-et -- (+) -- / \ -- x x ← két példány! -- Gráf: x+x megosztja x-et -- (+) -- / \ -- └── x ┘ ← egy csomópont, két él -- GHC (Haskell fordító) gráf újraírást használ! -- Így nem értékeli ki x-et kétszer: let x = expr in x+x
💡 LAZY + GRAPH
A lusta kiértékelés + gráf újraírás = minden részkifejezés max. egyszer értékelődik ki.
🏷️
Típus Tímea
TYPE SYSTEM  ·  Int, Bool, Maybe, [a], a->b

Tímea mindenre rábélyegez egy típust, és a fordító ellenőrzi, hogy a bélyegek stimmelnek-e. Haskellben a típusrendszer erős és statikus — ha lefordul, szinte biztosan helyes. A típus dokumentál is.

"A típusod árulja el, ki vagy. Hazudni nem lehet."
-- Típus annotáció: negyel :: Int -> Int negyel x = x * 4 -- Algebrai adattípus (ADT): data Szin = Piros | Zold | Kek -- Maybe: lehet érték, lehet semmi osztDa :: Int -> Int -> Maybe Int osztDa _ 0 = Nothing osztDa a b = Just (a `div` b)
💡 TYPE INFERENCE
Hindley-Milner: Tímea a legtöbbször maga kitalálja a típust — nem kell mindig írni.
🌯
Fold Frida
FOLD / REDUCE  ·  foldl, foldr

Frida összegöngyöl egy listát egyetlen értékké. Egy akkumulátorból és egy függvényből indulva végigmegy minden elemen és "összehajtogatja" őket. A sum, product, length mind Frida variánsa.

"Listát kapok. Értéket adok. A részleteket senki se lássa."
-- foldl (balról): ((0+1)+2)+3 foldl (+) 0 [1,2,3] -- → 6 -- foldr (jobbról): 1+(2+(3+0)) foldr (+) 0 [1,2,3] -- → 6 -- map Frida segítségével: maySajat f xs = foldr (\x acc -> f x : acc) [] xs -- length Fridával: lengthSajat = foldl (\acc _ -> acc+1) 0
💡 MINDEN FOLD
map, filter, sum, any, all — mind kifejezhető fold-dal.
